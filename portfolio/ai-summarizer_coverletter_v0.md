# 1. 어떤 문제를 해결할 때, 'Why?'를 놓치지 않고 문제의 본질을 찾는 데 집중했던 경험이 있다면 적어주세요. (1000자 이내)

저는 ai-summarizer 프로젝트를 개발하면서 API 키 노출이라는 문제를 겪었습니다. 처음에는 코드에 API 키를 직접 하드코딩했고, 기능은 동작했기 때문에 이 정도면 충분하다고 생각했습니다. 그러나 GitHub에 push한 직후 GitHub의 Push Protection이 API 키 노출을 감지해 차단했고, 저는 단순히 키를 지우는 것으로 끝내지 않고 'Why?'를 물었습니다.

왜 이 문제가 발생했는가를 추적하자, 근본 원인은 키를 코드와 같은 공간에 두는 설계 자체였습니다. 키를 지워도 git history에 남아있다는 사실도 이 과정에서 깨달았습니다. 단순 수정이 아닌 구조적 해결이 필요했습니다.

이를 위해 .env 파일로 키를 분리하고 .gitignore에 등록했으며, 이미 노출된 히스토리는 git-filter-repo로 완전히 제거했습니다. 또한 노출된 키는 즉시 재발급했습니다. 표면적 에러를 고치는 것을 넘어, 왜 이 설계가 취약했는지를 파악하고 구조 자체를 바꾼 경험이었습니다. 이후로는 새 프로젝트를 시작할 때 환경변수 분리를 가장 먼저 설정하는 습관이 생겼습니다.

# 2. 어떤 일을 하면서 '이 정도면 충분하다'는 생각 대신, 더 완성도를 높이고 싶어 끝까지 몰입했던 경험이 있나요? 그때 어떤 점이 가장 도전적이었고, 어떻게 극복하셨는지 적어주세요. (1000자 이내)

저는 ai-summarizer 프로젝트에서 파일 요약 기능을 개선하는 작업을 하면서 이 정도면 충분하다는 생각 대신 더 완성도를 높이고 싶어 몰입했던 경험이 있습니다. 초기에는 summarize_file 함수를 추가하고 파일을 읽어 요약을 생성하는 것이 목표였지만, 작업을 진행하면서 여러 가지 에러와 마주하게 되었습니다.

특히 FileNotFoundError와 ValueError가 발생했을 때, 단순히 에러를 수정하는 데 그치지 않고 왜 이러한 문제가 발생했는지를 깊이 고민하게 되었습니다. 이 과정에서 파일 경로와 파일의 존재 여부를 확인하는 것이 단순한 해결책이 아니라, 코드의 구조와 설계가 문제를 유발하고 있다는 사실을 깨달았습니다. 그래서 저는 파일을 읽는 방식을 재설계하고, 예외 처리 로직을 강화하여 안정성을 높이는 방향으로 접근했습니다.

또한 긴 텍스트를 효과적으로 요약하기 위해 chunk_text 함수를 구현하는 과정도 도전적이었습니다. 여러 번의 시행착오를 겪으면서도 문제의 근본 원인을 찾고 해결책을 모색하는 과정에서 많은 것을 배웠습니다. 결국 완성도를 높이는 데 필요한 것은 단순한 수정이 아니라, 문제의 본질을 이해하고 지속적으로 개선하려는 노력이라는 것을 깨달았습니다.

# 3. [Tech] 공고 내 '세부 업무'를 참고하여 Platform/Client/Server/Infra 를 관심 순으로 나열하고 (ex. Platform > Client = Server > Infra, 관심도가 유사하다면 모두 '='으로 작성), 각 분야에 관심을 가지게 된 계기 혹은 경험(ex. 스터디, 교육 등)이 있다면 서술해 주세요. (1000자 이내)

Server > Platform > Client > Infra

Server에 가장 큰 관심을 갖게 된 계기는 ai-summarizer 프로젝트에서 Groq API를 연동하는 과정이었습니다. 외부 API와 통신하고 응답을 파싱하며 에러를 처리하는 과정에서 서버 로직의 설계가 서비스 안정성에 얼마나 직결되는지 체감했습니다. 특히 FileNotFoundError, KeyError, 401 인증 오류 등을 단계적으로 해결하면서 견고한 서버 코드를 작성하는 것에 흥미를 느꼈습니다.

Platform에 대한 관심은 DevTrace 프로젝트를 통해 생겼습니다. bash_history, git log, inotify 등 리눅스 시스템 자원을 직접 다루며 플랫폼 레벨에서 데이터를 수집하고 파이프라인을 구성하는 경험이 매력적이었습니다. 개발자가 코딩에만 집중해도 시스템이 알아서 기록하는 구조를 설계하면서 플랫폼 개발의 재미를 알게 되었습니다.

Client는 ai-summarizer의 HTML 뷰어와 DevTrace 웹 대시보드를 구현하면서 관심이 생겼습니다. 사용자가 직접 보는 인터페이스가 경험 전체를 좌우한다는 것을 느꼈지만, 아직 Server와 Platform에 비해 깊이 있는 경험이 부족합니다.

Infra는 현재까지 직접 다룬 경험이 적어 관심도가 상대적으로 낮지만, 서비스를 운영 환경에 배포하는 과정에서 자연스럽게 배워가고 싶은 분야입니다.
